# 📊 Financial Tracker — Экзаменационный проект

> **Финансовый трекер** — веб-приложение для учёта личных доходов и расходов с серверным рендерингом, фильтрацией, пагинацией и экспортом данных.

---

## 🚀 Быстрый старт 

Если у вас **чистая база данных Oracle**, просто запустите приложение:

```bash
# 1. Убедитесь, что Oracle запущен и доступен на localhost:1521
# 2. В application.yml проверьте:
#    spring.sql.init.mode: always  (для авто-инициализации БД)

# 3. Соберите и запустите:
mvn clean spring-boot:run

# 4. Откройте в браузере:
http://localhost:8080
```

✅ Приложение автоматически:
- Создаст таблицы `categories`, `transactions`
- Создаст последовательность `transactions_seq`
- Заполнит справочник категорий (8 записей, включая «Другое» для доходов и расходов)
- Запустится на порту `8080`

## 🗄️ Настройка базы данных

### Вариант А: Автоматическая инициализация (рекомендуется)

В файле `src/main/resources/application.yml` убедитесь, что стоит:

```yaml
spring:
  sql:
    init:
      mode: always  # ← Скрипты выполнятся при старте приложения
      continue-on-error: true # ← Игнорировать ошибки при первом запуске (DROP несуществующих таблиц)
```

При запуске `mvn spring-boot:run`:
1. `schema.sql` создаст таблицы с последовательностью `transactions_seq` для генерации ID
2. `data.sql` через `MERGE` заполнит справочник категорий

### Вариант Б: Ручная инициализация

Если `mode: never`, выполните скрипты вручную:

1. **Запустите Oracle** (Docker или локально)
2. **Выполните скрипты вручную** через SQL Developer / DBeaver / SQL*Plus:

```sql
@src/main/resources/schema.sql  -- Создаёт таблицы, индексы, sequence
@src/main/resources/data.sql    -- Заполняет справочник категорий
```

**Схема БД включает:**
- `categories` — справочник категорий (8 записей)
- `transactions` — финансовые операции
- `transactions_seq` — последовательность для генерации ID
- 4 индекса для оптимизации частых запросов
- `CHECK`-ограничения на уровне БД

```

### 🔐 Настройки подключения (application.yml)

```yaml
spring:
  datasource:
    url: jdbc:oracle:thin:@localhost:1521/XEPDB1  # Для Oracle XE(Docker)
    username: USERNAME            
    password: PASSWORD                   
    driver-class-name: oracle.jdbc.OracleDriver
```

### 🔑 Принцип слоёв

```
Controller → DTO → Service → Repository → Database
```

- **Controller**: Приём HTTP-запросов, валидация входа, подготовка модели для View
- **Service**: Бизнес-правила, транзакции (`@Transactional`), вызов репозиториев
- **Repository**: Доступ к БД (Spring Data JDBC + JdbcClient)
- **Model/DTO**: Сущности БД и объекты передачи данных

## 🎮 Использование

### 📊 Дашборд

- Показывает доходы, расходы и баланс за выбранный период
- Фильтр: период + валюта (суммируются только операции в выбранной валюте)
- Кнопки быстрого добавления дохода/расхода

### 📋 Список операций

| Функция           | Как использовать                                                      |
|-------------------|-----------------------------------------------------------------------|
| 🔍 Фильтрация     | Выберите тип, категорию, период, введите текст комментария → «Фильтр» |
| 📄 Пагинация      | Кнопки ««» / «»» в нижней части таблицы                               |
| ✏️ Редактирование | Нажмите иконку карандаша в строке операции                            |
| 📦 Архивация      | Нажмите иконку коробки → операция исчезнет из активного списка        |

> 💡 **Подсказка**: Категории с одинаковым именем различаются по типу: `Другое (доход)` / `Другое (расход)`.

### ➕ Создание операции

1. Выберите тип: 📈 Доход / 📉 Расход
2. Введите сумму (> 0), выберите валюту (RUB/USD/EUR)
3. Укажите дату (не в будущем) и категорию
4. (Опционально) Добавьте комментарий до 500 символов
5. Нажмите «💾 Сохранить»

> ⚠️ **Валидация**: При ошибке форма не сбрасывается — данные сохраняются, подсвечиваются проблемные поля.

### 🗑 Корзина

- Просмотр архивированных операций
- ↩️ **Восстановить**: Вернуть операцию в активный список
- 🗑️ **Удалить навсегда**: Полное удаление из БД (необратимо!)

## 🎓 Вопросы для защиты

### 🔹 Архитектура

**В: Почему Spring Data JDBC, а не JPA?**
> *«Курс фокусируется на прозрачной работе с БД. JDBC даёт полный контроль над типами данных, генерацией ключей и оптимизацией запросов. Нет скрытого dirty checking, N+1 проблем и сложного кэширования первого уровня. Для экзаменационного проекта это оптимальный баланс между простотой и функциональностью.»*

**В: Как реализована пагинация?**
> *«Использую стандартный для Oracle синтаксис `OFFSET :off ROWS FETCH NEXT :lim ROWS ONLY`. Отдельным запросом считаю `COUNT(*)` с теми же фильтрами для расчёта общего числа страниц. Это позволяет эффективно работать с большими объёмами данных без загрузки всех записей в память.»*

### 🔹 Валидация

**В: Где и почему валидация в трёх местах?**
> *«1) DTO: синтаксическая проверка (формат, обязательность) — быстрая, до обращения к БД. 2) Service: бизнес-правила (соответствие типа категории, дата не в будущем) — требует данных из БД. 3) БД: ограничения `CHECK`, `FK`, `UNIQUE` — последняя линия защиты, гарантирует целостность даже при прямом доступе к БД.»*

### 🔹 Данные

**В: Как работает мягкое удаление?**
> *«Поле `status` со значениями `ACTIVE`/`ARCHIVED`. Все выборки фильтруют `WHERE status='ACTIVE'`. Архивация — это `UPDATE status='ARCHIVED'`, а не `DELETE`. Это позволяет восстанавливать данные и вести аудит. Физическое удаление — только из корзины, по явному запросу пользователя.»*

**В: Как реализована генерация первичных ключей?**
> *«Я использую явную последовательность Oracle transactions_seq. Перед вставкой новой записи сервис явно запрашивает следующее значение через SELECT transactions_seq.NEXTVAL FROM DUAL с помощью JdbcClient, устанавливает его в поле id сущности, а затем выполняет прямой INSERT через параметризированный запрос. Этот подход:
✅ Полностью прозрачен и контролируется кодом
✅ Не зависит от магии repository.save() и интерфейса Persistable
✅ Работает стабильно на любой версии Oracle (12c+)
✅ Легко тестируется и отлаживается
Для обновления и удаления существующих записей используется стандартный CrudRepository.save() / delete(), так как в этих операциях id уже известен и проблем с определением типа операции не возникает.»*

### 🔹 Тестирование

**В: Как тестируется работа с БД без Oracle?**
> *«Интеграционные тесты используют H2 in-memory database с отдельной схемой в `src/test/resources/`. Профиль `test` активируется через `@ActiveProfiles`. Тестовая схема адаптирована под H2: `BIGINT` вместо `NUMBER`, `AUTO_INCREMENT` вместо `SEQUENCE`. Это позволяет быстро запускать тесты без подключения к реальной Oracle.»*

**В: Какие тесты покрывают проект?**
> *«14 тестов: smoke-тест контекста, тесты репозиториев (фильтры, пагинация, агрегация), юнит-тесты сервисов с Mockito, тесты контроллеров через `MockMvcBuilders.standaloneSetup`. Покрыты все три слоя: Controller, Service, Repository.»*


### 🔹 Безопасность

**В: Как защищено от SQL-инъекций?**
> *«Все запросы через `JdbcClient` используют параметризацию: `:param` вместо конкатенации строк. Значения передаются отдельно от кода запроса, драйвер экранирует их автоматически. Это стандартная и надёжная защита.»*

**В: Что будет при параллельном обновлении одной записи?**
> *«Oracle заблокирует строку на время транзакции. При конфликте выбросится `CannotAcquireLockException`, который обрабатывается в `@ControllerAdvice` с дружественным сообщением пользователю. В `@Transactional` по умолчанию настроен rollback при любой ошибке.»*

### 🔹 Финансы

**В: Почему нельзя суммировать разные валюты?**
> *«Рубли, доллары и евро имеют разную покупательную способность. Сумма 1000 RUB + 50 EUR = 1050 финансово бессмысленна. Поэтому на дашборде добавлен фильтр валюты: пользователь явно выбирает, в какой валюте смотреть итоги. В списке операций валюта отображается рядом с суммой. Для агрегации в будущем можно добавить конвертацию по курсу.»*
